# Windows 与 macOS 桌面应用落地方案

## 历史记录

- 2026-07-21 liuxiaoke 创建桌面应用落地方案，明确 Tauri 2 技术路线、浏览器与桌面双运行时边界、数据恢复策略、文件读写、双平台打包和验收门槛。

## 1. 文档定位

本文用于指导后续 Codex 将现有 `heima-record` 本地 Web 赛事记录台扩展为 Windows 和 macOS 桌面应用。

本文是开发过程方案，不包含本次代码实现。后续实施必须遵守以下原则：

- 保留现有单文件 HTML 发布能力。
- Windows 与 macOS 共用现有赛事业务代码，不维护两套产品。
- 桌面能力只进入基础设施适配层，不侵入计分、编排、排名等领域逻辑。
- 先解决数据可靠性，再处理安装包外观和发布渠道。
- 每个阶段独立验证，失败时可以回退，不一次性混入全部改动。

## 2. 原始需求与问题本质

原始需求是：将已经实现的轻量 HTML 比赛记录台，提供为功能相同的 Windows 和 macOS 应用。

问题本质不是重新开发桌面端界面，而是让现有 Web 应用获得四项桌面能力：

1. 作为独立应用启动和安装。
2. 在操作系统应用数据目录中可靠保存赛事状态。
3. 使用原生文件打开、保存窗口处理 Excel、CSV 和 JSON。
4. 生成可交付、可验证、可签名的 Windows 与 macOS 安装包。

因此最短路径是增加跨平台桌面容器和运行时适配层，而不是复制现有代码后重写。

## 3. 当前工程事实

以下结论来自 2026-07-21 当前工程源码，后续实施前仍应重新核对：

| 现状 | 代码位置 | 对桌面化的影响 |
|---|---|---|
| React 18、TypeScript、Vite 单页应用 | `package.json`、`src/main.tsx` | 可直接作为 Tauri 前端，无需重写界面 |
| 赛事、计分、编排和排名逻辑位于前端领域模块 | `src/domain/` | 应完整复用，不进入 Rust 层 |
| 自动保存使用 IndexedDB | `src/services/storage.ts` | 浏览器继续使用；桌面需要独立持久化实现 |
| 文件导入以 Web `File` 对象为输入 | `src/services/importer.ts`、`playerImporter.ts`、`ruleConfig.ts`、`backup.ts` | 桌面读取字节后可转换为 `File`，继续复用解析器 |
| 文件导出通过 Blob 和浏览器下载链接 | `src/services/download.ts` | 桌面需要切换为原生保存窗口和文件写入 |
| 无 Node 交付包为单文件 HTML | `scripts/prepare-release.mjs` | 必须保留，不能被桌面构建替代 |
| 已有 lint、领域测试、Web 构建和发布包检查 | `package.json`、`scripts/verify-release.mjs` | 应成为桌面化后的基础回归门槛 |

当前业务能力包括选手与场次导入、赛事编排、现场计时计分、规则判定、申诉与撤销、赛后修正、排名签表、Excel/CSV 结果导出和 JSON 完整备份。桌面化不改变这些业务语义。

## 4. 技术决策

### 4.1 采用 Tauri 2

默认采用 Tauri 2 作为 Windows 和 macOS 桌面容器。

原因：

- 可以直接使用现有 Vite 构建结果。
- Windows 使用 WebView2，macOS 使用系统 WebKit，不需要在安装包内重复携带完整 Chromium。
- 支持原生文件对话框、受控文件系统访问、单实例运行和平台安装包。
- Rust 层只承担容器与系统能力，现有 TypeScript 业务逻辑继续保持单一来源。

### 4.2 不采用的方案

| 方案 | 不作为默认方案的原因 |
|---|---|
| Electron | 能复用 Web 代码，但携带 Chromium 和 Node，安装体积、内存占用更高，不符合当前轻量离线工具定位 |
| PWA | 可以安装图标，但安装、文件访问、发布和系统信任体验依赖浏览器，不等同于稳定的双平台桌面交付 |
| Windows/macOS 原生重写 | 需要维护多套 UI、领域逻辑和文件格式，实现成本高且容易产生规则差异 |

如果后续发现 WebView2 与 WebKit 对关键现场交互存在无法规避的不一致，再重新评估 Electron；不能仅因团队熟悉 JavaScript 就提前放弃轻量方案。

## 5. 目标架构

```text
现有 React UI
    |
    +-- 赛事领域逻辑 src/domain                保持不变
    +-- Excel/CSV/JSON 解析与生成 services      保持单一来源
    |
运行时适配层 src/platform
    |
    +-- Browser
    |     +-- IndexedDB
    |     +-- input file / Blob download
    |
    +-- Desktop
          +-- Tauri AppLocalData
          +-- 原生打开/保存窗口
          +-- 受控文件系统
          +-- 单实例窗口
```

边界要求：

- `src/domain/` 不得导入任何 Tauri API。
- Excel、CSV、JSON 的字段和内容生成不得分出桌面专用版本。
- 浏览器与桌面只在存储、文件选择、文件保存和运行时状态展示上分支。
- Rust 不计算比分、排名、晋级关系，也不解析比赛规则。

## 6. 运行时适配合同

后续实现应先定义合同，再接入 Tauri。

### 6.1 运行时识别

运行时模块只回答一个问题：当前是否处于 Tauri 桌面容器。

禁止通过用户代理、操作系统名称或窗口尺寸推断桌面运行时。

### 6.2 状态仓库

状态仓库至少提供以下能力：

| 能力 | 输入 | 输出 | 失败行为 |
|---|---|---|---|
| 加载赛事状态 | 无 | `TournamentState` | 尝试恢复文件；全部失败后返回明确错误，不静默覆盖 |
| 保存赛事状态 | `TournamentState` | 完成状态 | UI 显示失败并提示立即导出备份 |
| 创建恢复快照 | 当前状态 | 快照标识 | 不阻断主要保存，但记录错误 |

浏览器实现继续使用 IndexedDB；桌面实现使用 Tauri 应用本地数据目录。

### 6.3 文件访问

文件访问适配层分为两个方向：

- 打开：选择本地文件，读取字节，转换为现有解析器使用的 `File` 对象。
- 保存：接收现有导出器生成的 Blob、建议文件名和扩展名，通过原生保存窗口写入目标路径。

用户取消文件窗口是正常流程，不应显示为错误。

## 7. 桌面数据可靠性设计

### 7.1 为什么不能只保留 IndexedDB

IndexedDB 可以继续支持浏览器版，但桌面赛事工具需要明确的数据目录、损坏恢复和可解释的保存状态。仅依赖 WebView 内部存储会让排查、迁移和异常恢复都依赖具体 WebView 实现。

### 7.2 数据文件

桌面状态文件建议使用版本化信封，至少包含：

| 字段 | 含义 |
|---|---|
| `app` | 固定为 `heima-record`，防止误读其他 JSON |
| `version` | 桌面状态文件格式版本 |
| `savedAt` | 实际完成保存的时间 |
| `state` | 完整 `TournamentState` |

读取后仍必须执行现有 `normalizeState`，保证旧备份缺少新字段时可以补默认值。

### 7.3 原子保存与回退

推荐使用双槽状态文件：

```text
state/current-a.json
state/current-b.json
```

每次保存流程：

1. 将新状态写入目标槽位对应的临时文件。
2. 写入完成后通过同目录重命名替换目标槽位。
3. 只有替换成功后才切换下一次使用的槽位。
4. 启动时读取两个槽位，选择 `savedAt` 最新且结构有效的一份。
5. 最新文件损坏时自动回退另一份，不立即创建空白状态覆盖现场数据。

所有保存操作必须串行排队，避免计时器更新与现场按钮操作同时完成写入后发生旧状态覆盖新状态。

### 7.4 历史快照

建议默认策略：

- 首次成功保存创建一份快照。
- 此后状态发生变化时，每 5 分钟最多创建一份。
- 保留最近 20 份，超过后删除最旧快照。
- 快照文件名包含 UTC 时间，内容与状态信封一致。

这些数值是初始默认值，可以在现场数据量验证后调整，但不能取消容量上限。

### 7.5 Web 数据迁移

浏览器 IndexedDB 与桌面应用数据目录属于不同安全边界，不能假设桌面版会自动读取浏览器数据。

正式迁移流程：

```text
HTML 版导出完整 JSON 备份
-> 桌面版导入备份
-> 桌面版自动保存
-> 关闭并重启桌面版验证恢复
```

不得编写扫描浏览器数据目录的迁移逻辑。

## 8. 文件导入导出设计

### 8.1 导入

桌面端使用 Tauri 原生打开窗口，并按用途限制扩展名：

| 用途 | 允许扩展名 |
|---|---|
| 选手名单 | `xlsx`、`xls`、`csv` |
| 场次 | `xlsx`、`xls`、`csv` |
| 规则 | `xlsx`、`xls`、`csv` |
| 完整备份 | `json` |

读取文件字节后转换为 Web `File`，继续调用现有 `parsePlayerFile`、`parseMatchFile`、`parseRuleFile` 和 `parseStateBackup`。

注意：当前界面允许选择 `xls`，但 ExcelJS 对旧二进制 XLS 的真实支持需要单独验证。若当前 Web 版同样无法解析，应该修正文案或转换要求，不能只在桌面版增加另一套解析器。

### 8.2 导出

桌面端使用原生保存窗口，保留现有默认文件名和扩展名。

必须覆盖：

- 比赛结果 CSV。
- 比赛结果 Excel。
- 赛事结果 Excel。
- 分组结果 Excel。
- 规则 Excel。
- 完整备份 JSON。

浏览器版继续使用当前 Blob 下载方式，避免影响单文件 HTML。

## 10. Tauri 权限与安全

原则是默认拒绝，只开放实际需要的能力。

桌面主窗口只需要：

- 原生打开窗口。
- 原生保存窗口。
- 读取用户明确选择的文件。
- 写入用户明确选择的保存路径。
- 读写应用本地数据目录下的状态与快照目录。
- 创建目录、重命名临时文件、删除过期快照。

不应开放：

- Shell 命令执行。
- 任意主目录递归访问。
- 网络请求插件。
- 不受范围约束的文件系统权限。
- WebView 中的远程页面加载。

生产配置需要设置 CSP，只允许应用自身资源、Tauri IPC、Blob 和必要的内联样式。新增能力后应重新审查 CSP，不能直接设为永久关闭。

## 11. 单实例与窗口行为

桌面版必须启用单实例约束，原因是两个窗口同时自动保存同一赛事会产生不可预测的覆盖关系。

重复启动应用时：

1. 不创建第二个赛事进程。
2. 如果主窗口最小化则恢复。
3. 显示并聚焦现有主窗口。

默认窗口建议：1440 × 900，最小 1024 × 720。最终值可以根据真实比赛电脑复核。

## 12. 双平台构建与发布

### 12.1 Windows

建议首个正式目标为 Windows 10/11 x64。

交付物优先级：

1. NSIS `Setup.exe`：面向现场普通用户。
2. MSI：面向需要集中部署的环境，确认有需求后再作为强制交付物。

WebView2 策略：

| 模式 | 特点 | 使用条件 |
|---|---|---|
| 系统运行时/嵌入引导程序 | 安装包较小 | 已确认目标为正常维护的 Windows 10/11 |
| 离线安装程序 | 安装包明显增大 | 现场电脑可能没有 WebView2且安装时完全断网 |
| 固定运行时 | 体积最大，安全更新由应用负责 | 除非有严格兼容冻结要求，否则不采用 |

Windows 安装包应在 Windows 构建机或 Windows CI 上生成和验收。macOS 交叉编译只作为应急手段，不能代替真实 Windows 安装测试。

### 12.2 macOS

交付物：

- `.app` 应用包。
- `.dmg` 安装镜像。

架构：

- Apple Silicon `aarch64`。
- Intel `x86_64` 是否保留：待确认。
- 如两者都支持，优先生成 Universal 包，并在两类机器或相应 CI 环境验证。

正式对外分发需要 Apple Developer 身份、代码签名和 notarization。未签名构建只能用于内部技术验证，不能作为正式交付完成证据。

### 12.3 应用标识

以下信息发布前必须确认，因为发布后修改会影响签名、更新和应用数据目录：

- 产品正式名称：当前建议“黑马兵击记录台”。
- Bundle Identifier：待确认，不在未确认时写死为正式值。
- Windows Publisher：待确认。
- macOS Team ID：待确认。
- Windows 签名证书：待确认。
- 应用图标与版权主体：待确认。

## 13. 分阶段实施计划

### 阶段 D1：运行时边界与 Web 回归

目标：建立适配层骨架，不接入桌面写入。

任务：

- 定义运行时、状态仓库和文件访问合同。
- 保持浏览器实现行为不变。
- 补充针对运行时分流的测试。
- 确认单文件 HTML 仍可生成和打开。

验收：

- `npm run lint` 通过。
- `npm run test` 通过。
- `npm run build` 通过。
- `npm run release` 通过。
- 发布包固定文件与当前要求一致。

### 阶段 D2：Tauri 最小容器

目标：桌面窗口可加载现有应用，但暂不声明数据可靠性完成。

任务：

- 初始化 Tauri 2 工程。
- 配置窗口、CSP、最小权限和单实例插件。
- 增加 Windows/macOS 图标资源。
- 启动开发模式并验证页面渲染。

验收：

- macOS 实际窗口启动。
- Windows 实际窗口启动。
- 重复启动只保留一个实例。
- Web 版全部回归检查继续通过。

### 阶段 D3：桌面状态持久化

目标：桌面版关闭、异常中断和状态文件损坏后仍可恢复。

任务：

- 实现双槽原子保存。
- 串行化自动保存。
- 实现快照保留与清理。
- 实现损坏文件回退。
- 在 UI 显示保存中、成功、失败。

验收：

- 正常关闭、强制结束进程、重启电脑后数据恢复。
- 人为损坏最新槽位后回退上一有效状态。
- 写入失败时 UI 明确报警，不显示虚假成功。
- 连续计时和快速计分不会出现旧状态覆盖新状态。

### 阶段 D4：原生文件导入导出

目标：所有现有文件能力在 Windows 与 macOS 行为一致。

任务：

- 替换四类导入入口。
- 替换全部导出保存入口。
- 验证中文文件名、空格路径和非 ASCII 路径。
- 验证取消窗口不改变赛事状态。

验收：

- 选手、场次、规则和备份均能导入。
- CSV、各类 Excel 和备份 JSON 均能保存并重新读取。
- Windows Excel 打开 CSV 不出现中文乱码。
- 浏览器单文件版读写能力不回退。

### 阶段 D5：安装包与发布说明

目标：形成可交付安装包和完整使用说明。

任务：

- 在目标平台构建安装包。
- 确认 WebView2 交付策略。
- 配置 macOS 签名、公证和 Windows 签名。
- 更新 README、使用说明和修改记录。
- 明确 HTML 版与桌面版的数据迁移步骤。

验收：

- 在未安装开发工具的干净机器安装、启动、卸载。
- 离线启动和完整比赛流程通过。
- 安装升级后旧赛事数据仍可读取。
- 系统不显示未知来源或损坏应用警告；内部未签名包必须明确标识为非正式交付。

计时验收需要特别关注系统休眠。当前控制台按固定间隔递减剩余秒数，如果电脑休眠后间隔回调暂停，唤醒后的时间语义可能与真实经过时间不一致。桌面化阶段应先定义需求：休眠期间比赛是否视为暂停。未确认前不能擅自改为按墙上时钟扣减。
