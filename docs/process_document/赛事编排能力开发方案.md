## 1. 文档定位

本文用于规划 `heima-record` 从“单场计时记分工具”扩展为“赛事编排 + 计时记分 + 晋级排名”的本地赛事控制台。

本文是开发过程方案，重点回答：

- 当前计时记分工具如何接入赛事编排能力。
- 需要新增哪些数据模型和业务模块。
- 小组循环、瑞士轮、单败、双败、三败如何分阶段落地。
- 排名规则、出线规则、弃权、判罚中止和纪律扣分如何配置。
- 每个阶段做到什么程度才算可以验收。

## 2. 第一性原理分析

当前系统已经解决的是“单场比赛如何被记录”：场次导入、计时、计分、警告、申诉、赛后修正、结果导出。

赛事编排要解决的是“下一场比赛从哪里来，以及结果如何影响下一轮”。

因此新增能力不应该改造计时台本身，而应该在计时台上方增加赛事管理层：

```text
选手名单
-> 赛制配置
-> 自动生成比赛场次
-> 使用现有控制台记录单场结果
-> 锁定本轮结果
-> 计算排名 / 晋级 / 下一轮配对
-> 生成后续场次
-> 输出最终名次
```

核心原则：

- 单场比赛仍由现有 `Match` 和控制台执行。
- 编排模块只负责生成、推进和解释比赛关系。
- 赛事积分、场上比分、真实胜场、轮空、无赛胜必须分开记录。
- 排名规则必须可配置、可复核、可解释。

## 3. 能力边界

### 3.1 本阶段纳入

- 输入或导入所有选手。
- 选择赛制并配置参数。
- 自动生成比赛轮次或签表。
- 复用当前控制台完成每场计时记分。
- 根据比赛结果计算排名、出线和下一轮对阵。
- 支持小组出线人数配置。
- 支持排名规则优先级排序和开关。
- 支持纪律扣分配置，例如警告、黄牌、红牌扣除最终赛事积分。
- 支持淘汰赛季军赛。

### 3.2 暂不纳入

- 多设备实时同步。
- 在线报名系统。
- 复杂裁判排班。
- 自动场地时间优化。
- 与外部赛事平台双向同步。

这些能力都不是赛事编排的最短闭环，后续可以作为独立阶段处理。

## 4. 总体架构

建议新增 5 个业务模块。

### 4.1 选手管理模块

用途：

- 维护赛事参赛者。
- 为分组、种子、瑞士轮配对、淘汰签位提供基础数据。

核心字段：

| 字段 | 含义 |
|------|------|
| `playerId` | 选手唯一 ID |
| `name` | 选手姓名 |
| `club` | 俱乐部或单位 |
| `seed` | 种子序号，可为空 |
| `status` | 正常、退赛、未签到 |
| `groupName` | 小组名，可由系统生成 |

### 4.2 赛制配置模块

用途：

- 描述本次赛事采用哪种赛制。
- 配置每种赛制的轮数、晋级人数、排名规则和异常处理。

需要支持的赛制：

- 小组循环 + 淘汰赛
- 瑞士轮 + 淘汰赛
- 直接单败淘汰赛
- 双败淘汰赛
- 三败淘汰赛

### 4.3 编排引擎模块

用途：

- 根据赛制配置生成当前阶段的比赛。
- 根据已锁定结果生成下一轮比赛。

编排引擎只生成对阵，不直接修改单场比分。

### 4.4 排名与晋级模块

用途：

- 按配置计算小组排名、瑞士轮排名、淘汰阶段晋级和最终名次。
- 输出排名依据，方便现场复核。

### 4.5 赛事状态模块

用途：

- 管理赛事阶段：未开始、预赛中、预赛锁定、淘汰赛中、已结束。
- 管理轮次锁定和赛后更正。

## 5. 数据模型建议

### 5.1 TournamentEvent

表示一个完整赛事项目，例如“长剑公开组”。

关键字段：

| 字段 | 含义 |
|------|------|
| `id` | 赛事项目 ID |
| `name` | 项目名称 |
| `format` | 赛制类型 |
| `stage` | 当前阶段 |
| `players` | 参赛选手 |
| `formatConfig` | 赛制配置 |
| `rankingConfig` | 排名配置 |
| `rounds` | 已生成轮次 |
| `brackets` | 淘汰签表 |

### 5.2 TournamentPlayer

表示赛事内选手，不直接替代当前 `Competitor`。

原因：同一名选手在赛事层需要种子、排名、退赛状态、纪律扣分等信息；单场 `Competitor` 只关心红蓝双方展示信息。

### 5.3 TournamentRound

表示小组赛或瑞士轮的一轮。

关键字段：

| 字段 | 含义 |
|------|------|
| `roundNo` | 轮次编号 |
| `stageType` | group、swiss、bracket |
| `status` | 草稿、已发布、已锁定 |
| `matchIds` | 本轮生成的场次 ID |

### 5.4 RankingRuleConfig

表示排名规则开关和优先级。

建议结构：

```text
[
  { key: "eventPoints", enabled: true, priority: 1 },
  { key: "buchholz", enabled: true, priority: 2 },
  { key: "buchholzCut1", enabled: true, priority: 3 },
  { key: "realWins", enabled: true, priority: 4 },
  { key: "cappedAverageScoreDiff", enabled: true, priority: 5 },
  { key: "disciplineScore", enabled: true, priority: 6 },
  { key: "headToHead", enabled: false, priority: 7 },
  { key: "playoff", enabled: true, priority: 99 }
]
```

其中：

- `eventPoints`：赛事积分。
- `buchholz`：对手分。
- `buchholzCut1`：去最低对手分。
- `realWins`：真实胜场，不含轮空和无赛胜。
- `cappedAverageScoreDiff`：封顶后的场均净胜分。
- `disciplineScore`：安全与纪律指标。
- `headToHead`：相互胜负。
- `playoff`：附加赛。

用户要求这些规则支持开关和优先级排序，因此 UI 不应写死排名顺序。

### 5.5 DisciplinePointConfig

用于配置警告、黄牌、红牌等对最终赛事积分的影响。

示例：

| 事件 | 最终赛事积分影响 |
|------|------------------|
| 口头警告 | 0 |
| 黄牌 | -0.5 |
| 红牌 | -1 |
| 黑牌 | -3 |
| 判罚中止 | 按规则配置 |

说明：

- 纪律扣分不应直接等同于单场比分扣分。
- 单场比分用于比赛胜负；纪律扣分用于赛事排名或纪律排名。
- 是否计入总赛事积分、是否仅作为破同分指标，需要配置。

## 6. 排名规则设计

### 6.1 默认排名优先级

默认采用参考文档中的瑞士轮破同分顺序：

1. 赛事积分。
2. 对手分 Buchholz。
3. 去最低对手分。
4. 真实胜场数。
5. 封顶后的场均净胜分。
6. 安全与纪律指标。
7. 淘汰线附加赛。

但系统必须允许赛事组织者关闭或调整这些规则。

### 6.2 胜场优先和积分优先

系统支持两种常用配置：

- 积分优先：胜 3、平 1、负 0，默认推荐。
- 胜场优先：真实胜场优先，适合不允许平局或强调胜负记录的赛事。

底层实现上不需要写两套排名算法，只需要把 `eventPoints` 和 `realWins` 的优先级交换。

### 6.3 净胜分规则

净胜分可以参与排名，但默认不应排在对手分之前。

原因：

- 净胜分过早参与排名，会鼓励面对较弱对手时追求扩大比分。
- HEMA 不应鼓励为了净胜分继续高风险进攻。

建议默认使用“封顶后的场均净胜分”：

```text
单场净胜分 = 红方得分 - 蓝方得分
封顶净胜分 = min(max(单场净胜分, -cap), cap)
场均封顶净胜分 = 封顶净胜分总和 / 真实比赛场数
```

`cap` 应跟随武器和规则配置，默认可设为目标分的一半或固定值。

### 6.4 相互胜负

相互胜负支持作为可选破同分规则，但默认关闭。

原因：

- 两人同分时，相互胜负清晰有效。
- 多人同分时，相互胜负会形成复杂循环，需要额外小表计算。
- 用户已明确多人同分采用附加赛，系统应优先支持附加赛路径。

建议：

- 两人完全同分时，可启用相互胜负。
- 三人及以上完全同分且影响晋级线时，生成附加赛。

### 6.5 多人同分附加赛

当排名规则全部比较后仍无法决定晋级线，系统应生成附加赛。

附加赛规则：

- 只在影响晋级、淘汰签位关键位置或奖牌名次时触发。
- 附加赛作为真实比赛记录。
- 附加赛结果只用于解决该排名冲突，不反向重算已锁定轮次配对。

## 7. 弃权、轮空和判罚中止

### 7.1 轮空

按参考文档执行：

- 轮空获得与胜利相同的赛事积分。
- 不计真实胜场。
- 不记录真实比分。
- 不提供真实对手分。
- 瑞士阶段同一选手最多轮空一次。

### 7.2 无赛胜

无赛胜与正常轮空分开记录：

- 可获得赛事积分。
- 不计真实胜场。
- 不计真实交手。
- 是否影响纪律或对手分需要单独配置。

### 7.3 弃权

建议默认：

- 已完成比赛保留。
- 未开始比赛按无赛胜、取消比赛或判负处理，具体由赛事配置决定。
- 退赛后不参与后续瑞士轮配对。
- 淘汰赛签表开始后退赛，原则上按弃权处理，不重排签表。

### 7.4 判罚中止

当前系统已支持单场警告和判罚中止结果配置。赛事层需要进一步定义它如何进入排名：

| 判罚结果 | 单场胜负 | 赛事积分 | 真实胜场 | 纪律扣分 |
|----------|----------|----------|----------|----------|
| 对方胜 | 对方胜 | 胜方按胜，负方按负 | 胜方可计真实胜场 | 负方按纪律配置扣分 |
| 本人胜 | 本人胜 | 本人按胜 | 本人可计真实胜场 | 按纪律配置处理 |
| 平局 | 平局 | 双方按平 | 不计胜场 | 按纪律配置处理 |
| 手动判定 | 裁判选择 | 按最终选择 | 按最终选择 | 按纪律配置处理 |

是否“判罚胜”计真实胜场应可配置。默认建议：若比赛已经实际发生且产生裁判裁决，可计入真实胜场；无赛胜不计真实胜场。

## 8. 赛制实现方案

### 8.1 小组循环 + 淘汰赛

推荐作为第一阶段落地。

原因：

- 当前系统已经有 `groupName` 和按组展示。
- 小组循环可以一次性生成全部场次。
- 小组赛结束后只需要做一次排名和出线计算。
- 对本地单机工具最友好。

核心配置：

- 每组人数。
- 每组固定出线人数。
- 是否允许跨组补位。
- 全场晋级总人数。
- 排名规则。
- 是否生成季军赛。

生成逻辑：

```text
选手列表
-> 按种子蛇形分组或随机分组
-> 每组生成单循环场次
-> 记录全部小组赛
-> 锁定小组赛
-> 计算小组排名和跨组排名
-> 生成淘汰赛签表
-> 生成季军赛
```

### 8.2 瑞士轮 + 淘汰赛

推荐作为第二阶段落地。

原因：

- 瑞士轮需要动态配对和轮次锁定。
- 对成绩确认、重复对阵检查、轮空选择要求更高。

核心配置：

- 瑞士轮轮数。
- 晋级人数。
- 排名规则。
- 轮空规则。
- 同俱乐部回避是否启用。
- 是否生成季军赛。

生成逻辑：

```text
第 1 轮：按种子或随机生成
-> 记录结果
-> 锁定本轮
-> 根据积分和约束生成第 2 轮
-> 重复直到固定轮数结束
-> 计算最终排名
-> 生成淘汰赛签表
```

### 8.3 直接单败淘汰赛

推荐作为第一阶段或第一阶段后半同步实现。

原因：

- 模型简单。
- 可作为小组赛后的淘汰阶段基础。

要求：

- 支持非 2 的幂人数。
- 高种子轮空。
- 支持季军赛。
- 淘汰赛不得以平局结束。

### 8.4 双败淘汰赛

推荐第三阶段实现。

原因：

- 败者组来源复杂。
- 当前版本明确不启用决赛重置。
- 现场休息和等待链更复杂。

核心配置：

- 总决赛一场决胜。
- 败者组排序规则。
- 弃权和双负处理。
- 季军名次计算方式。

### 8.5 三败淘汰赛

推荐第四阶段实现，且默认隐藏在高级模式。

原因：

- 三败不适合作为常规公开赛默认赛制。
- 它的工程实现更像“累计负场淘汰池”，不是普通固定签表。

建议做法：

- 每名选手维护累计负场。
- 每轮对仍未达到 3 负的选手重新编排。
- 达到 3 负后退出争冠路径。
- 最终排名需要结合淘汰轮次、负场数和胜负记录。

## 9. UI 设计建议

### 9.1 新增页面

建议新增 4 个导航入口：

- 选手
- 编排
- 排名
- 签表

当前“场次”和“控制台”保留。

赛事工作台改造落地：

- 导航新增“首页”，独立“导入”入口移除，导入场次合并到“场次”。
- 首页展示赛制、参赛人数、阶段和比赛进度，并提供保留计分规则的赛事重置入口。
- 编排页集中维护赛事生成场次，仅允许编辑未开始场次的编号、现场分组和场地。
- 普通导入场次可在场次页按场或按组删除；赛事生成场次必须从编排页统一清空。
- 瑞士轮场地组只是现场执行分流，不形成独立积分池；配对、排名、轮次锁定和出线保持全局统一。
- 编排页新增“选手分组结果”：小组循环输出固定组名单，瑞士轮按轮次输出现场分组、对阵、场地和轮空。
- 分组结果从赛事选手和实际场次派生，不额外持久化副本，人工调整现场字段后展示与 Excel 自动同步。

### 9.2 典型流程

```text
导入选手
-> 选择赛制
-> 配置排名规则
-> 生成预赛
-> 到场次页/控制台记录比赛
-> 锁定轮次
-> 查看排名
-> 生成淘汰赛
-> 记录淘汰赛
-> 生成最终结果
```

### 9.3 排名配置 UI

排名规则建议做成可拖拽排序或上下移动排序：

| 启用 | 优先级 | 规则 | 参数 |
|------|--------|------|------|
| 是 | 1 | 赛事积分 | 胜 3 / 平 1 / 负 0 |
| 是 | 2 | 对手分 | 真实对手 |
| 是 | 3 | 去最低对手分 | 去 1 个 |
| 是 | 4 | 真实胜场 | 不含轮空 |
| 是 | 5 | 封顶场均净胜分 | cap = 5 |
| 是 | 6 | 纪律指标 | 黄牌、红牌扣分 |
| 否 | 7 | 相互胜负 | 仅两人同分 |
| 是 | 99 | 附加赛 | 晋级线触发 |

## 10. 分步推进计划

### 阶段 1：选手导入 + 小组循环 + 单败淘汰

目标：

- 形成第一个完整编排闭环。
- 复用现有计时记分控制台，不重写单场比赛执行层。

交付内容：

- 选手导入。
- 粘贴录入选手，每行格式为“姓名，单位，种子序号”。
- 小组自动生成。
- 小组循环场次自动生成。
- 小组排名计算。
- 小组出线人数配置。
- 单败淘汰签表生成。
- 季军赛生成。
- 最终名次导出。

实施边界：

- 编排层新增赛事状态，不替代现有 `Match`。
- 小组赛和淘汰赛都生成现有 `Match`，继续使用当前控制台计时记分。
- 小组分配按种子蛇形分组；没有种子时按录入顺序补默认种子。
- 第一阶段排名默认按赛事积分、真实胜场、净胜分、纪律扣分排序。
- 排名规则可配置化放入第二阶段，但第一阶段保留独立排名数据结构。
- 淘汰赛按当前排名生成初始签表，非 2 的幂人数给高种子轮空。
- 半决赛完成后推进时生成决赛和季军赛。

验收标准：

- 输入 12 名选手，可自动分为 2 个 6 人小组。
- 每组自动生成 15 场循环赛。
- 记录完小组赛后能计算排名。
- 每组前 N 名或全场前 Q 名可进入淘汰赛。
- 淘汰赛结束后生成冠军、亚军、季军、第四名。

### 阶段 2：排名规则配置化

目标：

- 让赛事组织者可配置排名优先级和纪律扣分。
- 将排序逻辑从固定代码判断改为赛事配置驱动。

交付内容：

- 排名规则开关。
- 排名规则优先级调整。
- 赛事积分配置。
- 警告、黄牌、红牌等纪律扣分配置。
- 多人同分附加赛生成入口。

实施边界：

- 排名规则使用“启用 / 停用 + 上移 / 下移”完成配置，不做拖拽。
- 附加赛规则保留为兜底项，不参与普通规则排序。
- 赛事积分支持配置胜、平、负、双负分值。
- 纪律扣分支持按警告级别配置扣除分值，并可选择是否直接计入赛事积分。
- 相互胜负作为可选规则参与排序；多人同分仍优先提示并生成附加赛。
- 附加赛第一版按同组同分选手生成单循环附加赛场次，继续复用现有控制台记分。
- 排名页展示“需要附加赛”标记，便于现场复核晋级线争议。

验收标准：

- 可将真实胜场排到赛事积分之前。
- 可关闭净胜分。
- 可启用或关闭相互胜负。
- 红牌扣赛事积分后，排名结果发生可解释变化。
- 晋级线完全同分时提示生成附加赛。

### 阶段 3：瑞士轮 + 单败淘汰

目标：

- 支持固定轮数瑞士轮主推荐赛制。

交付内容：

- 首轮按种子配对。
- 后续轮次按积分组配对。
- 禁止瑞士阶段重复交手。
- 奇数人数轮空。
- 轮次锁定。
- 瑞士轮最终排名。
- 晋级淘汰赛。

验收标准：

- 输入 20 名选手，配置 5 轮瑞士轮。
- 每轮完成后才能生成下一轮。
- 不出现重复对阵。
- 奇数人数时轮空不计真实胜场。
- 瑞士轮结束后可生成 12 人晋级淘汰签表。

实施落地：

- 赛制配置新增“小组循环 + 单败淘汰”和“瑞士轮 + 单败淘汰”选择，同一个交付包内切换，不新增独立发布包。
- 瑞士轮配置支持轮数、晋级人数、是否尽量避开同单位、奇数人数是否允许轮空。
- 瑞士轮首轮按种子顺序编排，后续轮次按当前排名顺序配对。
- 配对过程优先同积分段，尽量避开同单位，并检查瑞士轮历史对阵，避免重复交手。
- 每个瑞士轮有“已发布 / 已锁定”状态，只有本轮所有场次结束后才能锁定。
- 下一轮必须在当前轮锁定后生成，达到配置轮数后赛事进入“瑞士轮完成”状态。
- 轮空只增加赛事胜场积分，不计入真实胜场、净胜分或相互胜负。
- 瑞士轮排名复用第二阶段的排名规则配置，晋级人数使用“瑞士轮晋级人数”字段。
- 瑞士轮完成后复用现有单败淘汰签表和季军赛生成能力。
- 瑞士轮配置增加“首轮随机配对（忽略种子）”，默认关闭以保持原种子编排兼容性。
- 开启随机首轮时，奇数人数仍先按最低顺位且未轮空原则确定轮空选手，其余选手使用 Fisher-Yates 洗牌后两两配对。
- 随机只在首轮场次生成时执行一次，结果随赛事状态持久化；第二轮起继续使用排名驱动的瑞士配对规则。

### 阶段 4：直接单败与双败（三败暂缓）

目标：

- 完成直接单败和双败主流程，三败不进入当前交付路线。

交付内容：

- 直接单败淘汰赛。
- 双败淘汰赛。
- 三败淘汰赛（原计划，现暂缓）。
- 无决赛重置总决赛。
- 负场累计和最终排名。

验收标准：

- 任意人数单败可生成正确轮空和季军赛。
- 双败能正确维护胜者组、败者组和一场总决赛。
- 三败累计 3 负淘汰验收暂不执行。

阶段 4A 实施落地：

- 先交付“直接单败淘汰赛”独立入口，不在本阶段混入双败和三败。
- 赛制配置新增“直接单败淘汰赛”，该赛制不显示小组人数、瑞士轮轮数、预赛晋级人数等无关字段。
- 直接单败以选手名单中的种子序号作为签表排序依据，未填写种子时按姓名兜底排序。
- 直接单败不经过预赛排名，所有正常参赛选手都直接进入淘汰签表。
- 签表生成继续复用现有单败淘汰逻辑，非 2 的幂人数自动给高种子轮空。
- 季军赛开关继续复用现有配置，在半决赛后自动生成。
- 直接单败生成时会替换旧赛事编排场次，保留非赛事导入场次，避免不同赛制场次混杂。
- 排名页在直接单败赛制下提示“不需要预赛排名”，现场结果以签表推进为准。

阶段 4B 实施落地：

- 新增“双败淘汰赛”赛制入口，不在本阶段实现三败淘汰。
- 双败淘汰赛不做决赛重置；胜者组冠军和败者组冠军进行一场总决赛，胜者直接成为最终冠军。
- 首轮按选手种子生成胜者组签表，非 2 的幂人数自动给高种子轮空。
- 推进时从已完成签表节点重新计算选手负场数，0 负进入胜者组，1 负进入败者组，2 负视为淘汰。
- 所有当前待赛节点完成后才能生成下一波胜者组 / 败者组场次，避免同一选手同时出现在多个待赛场次。
- 败者组人数为奇数时，优先给高种子轮空，轮空不增加负场。
- 签表页展示胜者组、败者组和总决赛节点，比赛仍复用现有计时记分控制台。
- 排名页在双败淘汰赛制下提示“不需要预赛排名”，现场结果以签表推进为准。

### 10.5 编排配置和场地调度增强

本阶段将原瑞士轮专属配置提升为全赛制公共能力：

- 新赛事默认忽略种子，勾选“采用种子编排”后，小组循环、瑞士轮首轮、直接单败和双败首轮统一按种子工作。
- 随机抽签只在生成动作中执行一次，生成结果写入赛事状态；刷新和关闭浏览器不会重新抽签。
- 小组循环增加同单位规避开关。组容量是硬约束，同单位规避是优化目标；无法完全规避时保留容量并报告剩余冲突。
- 场地数量改为全赛制公共配置。小组赛以完整小组为调度单位，按组内场次数由大到小放入当前负载最低场地；其他阶段逐场补入当前负载最低场地。
- 瑞士轮继续使用全局排名，场地仅决定当轮现场分组，不改变配对和晋级逻辑。
- 编排结果 Excel 输出场次明细和场地统计，作为现场排程、打印和人工复核入口。
- 旧备份从瑞士轮随机开关和场地组数迁移到新公共字段，避免升级后历史赛事语义反转。

## 11. 推荐优先级

建议按以下顺序开发：

1. 小组循环 + 单败淘汰。
2. 排名规则配置化。
3. 瑞士轮 + 单败淘汰。
4. 直接单败独立入口。
5. 双败淘汰。
6. 三败淘汰（暂缓，不进入当前交付路线）。

原因：

- 小组循环最贴近当前数据模型，最快产生完整闭环。
- 排名规则配置是所有赛制共用基础，应在瑞士轮前完成。
- 瑞士轮依赖轮次锁定和动态配对，适合放在排名基础稳定之后。
- 双败和三败复杂度高，不应阻塞主流程上线。

## 12. 风险与控制点

### 12.1 排名错误风险

控制方式：

- 每次排名输出计算依据。
- 排名锁定前提供复核期。
- 锁定后修改必须留痕。

### 12.2 配对错误风险

控制方式：

- 瑞士轮重复对阵检查。
- 轮空次数检查。
- 人工调整必须记录原因。

### 12.3 现场误操作风险

控制方式：

- 生成下一轮前必须锁定上一轮。
- 已发布轮次修改需要二次确认。
- 淘汰签表开始后不允许整体重排，只允许局部弃权处理。

### 12.4 功能范围膨胀风险

控制方式：

- 第一阶段只做小组循环和单败。
- 瑞士轮、双败、三败不抢第一阶段范围。
- 多设备同步、裁判排班和在线报名另开阶段。
